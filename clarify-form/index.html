Problem Clarification Form

แบบฟอร์มถาม User ก่อนเริ่มสั่งพัฒนาระบบ

ใช้เป็น prerequisite เพื่อ clarify ปัญหา ลูกค้า impact scope constraints และ acceptance criteria ก่อนให้ AI หรือทีม dev เริ่มออกแบบ/เขียนระบบจริง

PurposeReduce wrong builds
Use beforePRD / UX / Architecture
OutputProblem brief
OwnerFounder / PM

1. ควรออกแบบเป็น Web Form ไหม?

ควรครับ โดยเฉพาะถ้าจะใช้ในบริษัทจริง เพราะช่วยบังคับให้ทุก idea ผ่านคำถามชุดเดียวกันก่อนใช้เวลา engineering

GoodStandardize intake

ทุก request ถูกเก็บด้วย schema เดียวกัน ทำให้เปรียบเทียบ จัด priority และส่งต่อให้ AI/team ได้ง่าย

GoodReduce ambiguity

ลดปัญหา “สั่งกว้างเกินไป” เช่น build dashboard, make AI app, improve onboarding โดยไม่มี context

Cautionอย่าถามเยอะเกิน

ควรมี short form สำหรับ idea ใหม่ และ deep form สำหรับ request ที่จะเข้า delivery จริง

Recommendation: ใช้ 2-stage form: Stage A Intake 8-10 คำถามเพื่อคัดกรอง และ Stage B Deep Clarification 25-35 คำถามเมื่อ idea ผ่าน gate แล้ว

2. Question Bank

คำถามเหล่านี้คือ prerequisite ก่อนเริ่ม PRD/UX/Architecture/Implementation

A. Problem

ปัญหาคืออะไร? ใครเจอ? เกิดตอนไหน? ปัจจุบันแก้ยังไง? ทำไมวิธีเดิมไม่พอ?

B. Customer & User

user หลักคือใคร? buyer คือใคร? มี segment ไหนเริ่มก่อน? ใครไม่ใช่ target?

C. Impact

เสียเวลา/เงิน/โอกาสเท่าไหร่? frequency? severity? ถ้าไม่แก้จะเกิดอะไร?

D. Workflow

workflow ปัจจุบันเป็นอย่างไร? จุดเจ็บอยู่ตรงไหน? ต้อง integrate กับเครื่องมืออะไร?

E. Success

ถ้าสำเร็จ user จะทำอะไรได้? metric ไหนดีขึ้น? acceptance criteria คืออะไร?

F. Scope

MVP ต้องมีอะไร? อะไรไม่ทำ? deadline? budget? constraints?

G. Risk

ข้อมูล sensitive ไหม? compliance? security? failure mode ที่รับไม่ได้คืออะไร?

H. Evidence

มีลูกค้าจริง/ข้อความ/support ticket/data/competitor proof อะไรบ้าง?

Gateต้องตอบได้ก่อนผ่านถ้ายังตอบไม่ได้
Discoveryใครเจอปัญหา และหลักฐานคืออะไรไปสัมภาษณ์/เก็บ evidence เพิ่ม
PRDgoal, non-goal, user journey, acceptance criteriaห้ามเริ่ม architecture
Architecturedata, integration, auth, scale, security constraintsทำ technical discovery/spike ก่อน
Implementationscope แตกเป็น issue เล็ก + test planแตก delivery plan ใหม่
