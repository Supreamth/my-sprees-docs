1. หลักการสำคัญ

ถ้าจะสร้างบริษัทจริง อย่าเริ่มจากการให้ AI เขียน code ทันที ให้เริ่มจากความชัดของปัญหา ลูกค้า และ acceptance criteria ก่อน

01Problem before code

เริ่มจากปัญหาและลูกค้า ไม่ใช่ feature list หรือ framework

02Artifact-driven

ทุก phase ต้องมีเอกสารหรือ artifact ที่ตรวจ ตัดสินใจ และส่งต่อได้

03Maker ≠ Checker

Model ที่เขียนงานไม่ควรเป็น model เดียวกับที่ approve งานตัวเอง

04Human decision owner

AI ช่วยคิดและทำ แต่ strategy, security, release และ scope ต้องมีคนรับผิดชอบ

05Acceptance criteria required

Requirement สำคัญต้องวัดได้ ทดสอบได้ หรือสังเกตได้

06No release without rollback

Production ต้องมี monitoring, incident owner และ rollback/containment plan

2. AI Model Roles

เลือก model ตามงาน ไม่ใช่ใช้ model เดียวทำทุกอย่าง บริษัท AI-native ที่ดีควรแยกบทบาทระหว่าง strategist, researcher, product, architect, builder, reviewer และ security

Strategist

Company thesis, positioning, business model
strongest reasoning model

Researcher

Market, competitors, APIs, vendors
web/research-enabled model

Product Manager

PRD, requirements, scope, roadmap
long-context reasoning model

UX Designer

User flows, wireframes, copy, states
design/vision-capable model

Architect

Architecture, data model, API contracts
strong reasoning model

Builder

Implementation, tests, refactor
coding-specialized model

Reviewer

Spec compliance, code review, QA review
separate reviewer model

Security Reviewer

Threat model, auth, permission, data risk
security-focused reasoning model

SRE / Analyst

Deploy, monitoring, metrics, feedback
DevOps + data-analysis models

Rule: อย่าให้ model ที่สร้างงานเป็นคนอนุมัติงานตัวเอง โดยเฉพาะ code, architecture, security และ release

6. AI Operating Rules

กฎสำหรับใช้ AI ในบริษัทจริงเพื่อให้ได้ velocity โดยไม่เสีย control

  1. Separate maker and checker: builder model เขียน, reviewer model ตรวจ, human approve เรื่องสำคัญ
  2. Use model by task: strategy ใช้ reasoning, research ใช้ web, code ใช้ coding, security ใช้ security-focused
  3. Never skip artifacts: ถ้าไม่เขียนเป็นเอกสาร ทีมจะไม่ aligned
  4. No production without rollback: ทุก release ต้องมีทางถอยหรือ containment plan
  5. No requirement without acceptance criteria: ถ้าวัดไม่ได้ ยังไม่พร้อม implement
  6. Record decisions: ใช้ ADRs และ decision log เพื่อไม่ต้องถกเรื่องเดิมซ้ำ
  7. Small PRs: ลด risk, review ง่าย, rollback ง่าย
  8. Human owns risk: AI ช่วยเสนอ แต่คนรับผิดชอบ security, legal, release และ pricing
